“What the Fuck have I Done Lately 🙈?”

细胞自动机规则

在一个棋盘上有若干细胞，用黑色代表细胞存活，白色代表细胞死亡，它们在棋盘上根据一定的规则重生或者死亡，规则如下：

  • 死亡：如果一个细胞的邻居活着的数量>3或者<2，则该细胞死亡
  • 重生：如果一个细胞的邻居活着的数量=3，则该细胞重生
  • 如果有其他情况则保持现状

程序逻辑分析

数据与表现分离

  • 程序业务逻辑与表现无关
  • 表现可以是图形的也可以是文本的
  • 表现可以是当地的也可以是远程的

View类与Field类的关系

两者的关系本质上是表现与数据的关系。在功能上来说，View只负责根据Field的内容画出相应的图形，而Field中负责数据的存放。为不断更新细胞的存活情况，再每一次的数据更新后，都需要调用repaint()接口通知View重新画出整个画面，不去考虑某个精确的局部更新，而是去全局更新，这样简化了程序设计的逻辑，从而降低了实现的难度，当然，这以“时间换简便”的思路，是建立在如今计算机运算速度显著提高的基础之上的。

责任驱动的设计

将某一程序要实现的某一功能，合理的分配到最合适的类或对象中去实现，是初期在程序设计中，非常重要的一环。

网格化

本程序调用了JFrame中的图形界面，对于一某图形界面本身，有较多的解析度，也就相应的有着更多的复杂度，但将图形界面网格格式化后，数据便有了确切对应的坐标，使得更容易处理。

关于程序的几个问题

问题零，为什么不是在Cell中提供setAlive(boolean)函数，而是采用较复杂的die()和reborn()？

答：对于这一问题，我没有明确的思路，毕竟两者没有时空复杂度上的区别，我目前能想到的改后的缺陷就是要传入的参数增多，代码结构复杂，可读性略微下降，但感觉都不算什么致命伤……

问题壹，为什么Field.getNeighbour()不直接通过Cell.isAlive()来返回周围存活细胞的数量，而是返回一个数组让外界来实现数量的统计？

答：Field只是Cell的一个简单的容器，功能只是存储Cell对象的集合，但若由Field类来实现统计，则需调用Cell类中的接口，提高了两类的耦合度，违背了设计原则。

问题贰，为什么不是由Cell本身来判断周围邻居的存亡情况来决定接下die()或reborn()的行为？

答：如果由Cell判断自己下一时刻的状态，那势必 要从Field中获取周围细胞的数据和Field的大小来判断自身的位置，这样就需要定义Field对象，这使得Cell类与Field类的耦合度升高，违背了设计原则。

我对于本程序的看法

在本程序的根本代码逻辑实现中，即在Cellmachine.java中，首先初始化了网格和细胞，其次在后续的更迭过程中，每次都是从左上一直到右下一格格扫过，对每一格细胞的下一刻的生存状态进行判断后，改变当前细胞的生存状态，再对下一格的细胞进行如上操作，但如此行列由先后顺序的判断操作，会导致上一个细胞的状态改变影响到下一个细胞本应该的进行的状态改变，算是该程序逻辑上的漏洞了，相应的反例也很容易举出，便不再赘述。

关于相应的解决方案，较简单的就是Copy一份相同的Field，将每个位置的Cell的下一时刻的生存状态记录到新的Field中，最终用新的Field通过View展示出来，最终所体现的结果就是整体在同一时刻一起改变，而不是有先后的时间顺序。

目前感觉，Java和C++一样，主要是处理面向对象的程序设计，有一定的框架就可以随写随想，相对较为自由，但某些类的接口与数据成员的关系的处理上，降低其耦合度和依赖性的设计较为困难。

以上看法纯个人浅薄见解，如有错误，请包涵斧正。
